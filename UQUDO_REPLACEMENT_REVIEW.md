# Uqudo Replacement Review

**Date**: April 1, 2026  
**Project**: eNROLL Android SDK  
**Current Vendor**: Innovatrics DOT  
**Candidate Vendor**: Uqudo  

---

## Executive Summary

This review evaluates replacing Innovatrics with Uqudo in the current eNROLL Android SDK.

The main conclusion is:

- Uqudo appears to cover the main features currently used from Innovatrics: document scan/OCR, face session/liveness, and passport NFC reading.
- The biggest migration effort is not Android camera UI. The biggest effort is backend integration because Uqudo is session- and JWS-based.
- The safest migration path is a phased replacement that keeps the current `eNROLL` public SDK API and current business flow shape, while adapting backend contracts behind the scenes.
- The current selfie video is generated by Innovatrics, not by our own code. This is an important migration risk because Uqudo documentation clearly describes face results and audit resources, but it does not clearly confirm the same raw exported liveness video contract used today.

Recommended PO message:

- Uqudo looks viable on paper for the features we actively use.
- This is mainly a backend/security integration project, not only an Android SDK swap.
- Safe estimate is a 1-week spike first, then around 6-8 calendar weeks for delivery if Android and backend work in parallel.

## What The App Does Today With Innovatrics

### Document flows

The current Android SDK does not rely on Innovatrics to finish the business process by itself.
Instead:

1. Innovatrics captures document images.
2. The app converts the image to base64.
3. The app uploads the image to current Luminsoft APIs.
4. Backend returns `CustomerData` or passport data already shaped for the current UI.

## Main Differences

| Area | Current with Innovatrics | Target with Uqudo | Impact |
| :--- | :--- | :--- | :--- |
| Document OCR | Mobile captures image, backend does business OCR/result handling through current APIs | Mobile runs Uqudo scan/session flow and returns signed result | Backend must validate and map Uqudo output into current `CustomerData` shape |
| Selfie / liveness | Mobile uploads selfie image and Innovatrics-generated `record` to current backend | Mobile launches Uqudo face session and receives signed face result | Backend must redesign face contract and possibly replace `record` workflow |
| Face auth / update | Mostly image upload to existing backend APIs | Backend must create face session first, then validate Uqudo result | More orchestration and session management |
| NFC / ePassport | Backend receives detailed vendor-specific mapped NFC payload | Backend must parse Uqudo reading result and remap to current downstream contract | High parity and testing effort |
| Security model | Mostly app/backend controlled | Token-based, session-based, signed JWS result model | Major backend/security change |
| Image handling | Mobile holds local bitmap/files directly | Uqudo commonly returns image IDs/resources retrievable through Info API | Backend may need extra image retrieval steps |
| SDK initialization | Local Innovatrics license and direct module initialization | Uqudo repo, dependency, token/session setup | Android change is moderate, backend change is larger |

## Backend Impact Table

| Backend work item | Why it is needed | Expected effort |
| :--- | :--- | :--- |
| Uqudo access token issuance | Android cannot safely own vendor credentials for session-based flows | High |
| Session creation for face flows | Uqudo face flow depends on server-created face sessions | High |
| JWS validation and parsing | Uqudo explicitly expects backend-side validation/parsing of results | Very high |
| Mapping Uqudo scan/read/face results to current DTOs | Keeps existing Android screens and flow logic stable | Very high |
| Image retrieval via Info API | Uqudo uses image/resource IDs that may need download before returning current payload shape | Medium-high |
| NFC data parity mapping | Current backend contract expects rich passport/MRZ/authentication payloads | Very high |
| Error mapping | Current UI already expects current backend failures and messages | Medium |
| Compliance review for liveness evidence | Current backend expects `record`; Uqudo raw-video parity is not yet confirmed | Very high |

### Why backend is the long pole

Backend is the long pole because the migration is not just replacing a mobile capture SDK. It also changes:

- how vendor credentials are used
- where trust is established
- how results are validated
- what payload shape comes back from the vendor
- how face and NFC data must be mapped into current existing APIs

## Video Ownership And Migration Risk

### Was the current video generated by us or by Innovatrics?

The current video content is generated by **Innovatrics**, not by our own code.

Our code only:

1. reads the vendor result bytes
2. base64-encodes them
3. writes them temporarily to a file
4. forwards them to backend as `record`

### Why this matters

This is one of the top migration risks.

Current backend behavior depends on receiving a raw `record` payload. Uqudo docs reviewed for this investigation clearly describe face results and image/audit resources, but they do not clearly confirm a raw exported liveness video contract that matches the current Innovatrics `record` usage.

This must be confirmed with Uqudo before committing to the final backend design.

## Top Risks

| Risk | Why it matters | Severity |
| :--- | :--- | :--- |
| Raw liveness video parity not confirmed | Current backend expects Innovatrics `record` | Critical |
| Backend architecture change | Uqudo requires token/session/JWS model | Critical |
| NFC parity | Current NFC request is rich and vendor-specific | High |
| Tenant feature availability | Uqudo access may be permission-gated per tenant/feature | High |
| Real-device behavior | Passport NFC can vary significantly by handset and passport | High |
| Image retrieval windows | Uqudo resource TTL may require timely backend retrieval | Medium |
| Error semantics mismatch | Existing UI expects current backend response and error patterns | Medium |

## Recommended Migration Strategy

Recommended strategy is a **phased swap**:

1. Keep the current `eNROLL` public API unchanged.
2. Keep the current onboarding/auth/update flow order unchanged.
3. Replace Innovatrics-dependent Android capture and reading activities.
4. Add a backend adapter layer that converts Uqudo session results into the current Luminsoft response models.
5. Only consider a full Uqudo-native redesign after the phased migration proves stable.

This reduces Android screen churn and puts the main change where it belongs: backend validation, mapping, and orchestration.
